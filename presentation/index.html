What Is System Integration?
No prior IT knowledge needed
04
Connecting Separate Systems into One
Three isolated apps → one unified platform
Definition

System Integration

Linking separate software so they share data and work as one — even if built by different teams.

Analogy — Universal Remote

One Remote, Three Devices

TV + Cable + Sound System — three devices, one remote. Our unified frontend is that remote for three university systems.

Analogy — School ID

One ID, Multiple Departments

Library, registrar, canteen — one card, works everywhere. Integration does this digitally.

✗ Before
Admission — Data stays here only
Registration — Re-enter data manually
Scheduling — No link to admissions
✓ After
Admission — Approved → data sent automatically
Registration — Account auto-created
Scheduling — Sees enrolled students instantly

Key Terms — Integration Concepts
How systems communicate in our project
API

Application Programming Interface

Rules that let one software request data from another. Like a waiter — takes your order to the kitchen, brings back your food.

GET /student/status/123 → returns status

Webhook

Automatic Notification

System A automatically notifies System B when something happens — no polling needed. Like a doorbell.

Admission approves → Registration notified instantly

REST API

Standard API Format

Uses HTTP verbs: GET read · POST create · PUT update · DELETE remove. All our systems use REST.

iFrame

Website Inside a Website

Embeds one site inside another. React loads Admission & Registration as iFrames — users see one page, not three separate URLs.

API Gateway

Central Request Router

Single entry point — React talks to the gateway, gateway forwards to the right backend. Our Node.js Express server.

CORS

Cross-Origin Security Rule

Browsers block cross-domain requests by default. React (port 3000) vs systems (port 80) = blocked — until we enabled CORS middleware.

Key Terms — Technologies & Patterns
The actual tools each system was built with
Laravel

PHP Web Framework

Builds web apps in PHP with routing, DB, and auth built-in. Used by Admission and Registration (Laravel 12).

React

JavaScript UI Library

Builds interactive interfaces. Our unified frontend dashboard — the first page all users see (React 19).

MySQL / PostgreSQL

Relational Databases

MySQL → university_db (Admission + Registration). PostgreSQL → Scheduling cloud deployment via Neon.

MVC

Model – View – Controller

Code in 3 layers: Model (data) · View (UI) · Controller (logic). Both Laravel systems follow this strictly.

Authentication & Session

Login & Stay Logged In

Auth = who are you? Session = remember you're logged in. Each system has its own — key reason shared login is hard.

X-API-Key

API Security Token

Secret key in HTTP headers proving the caller is authorized. Webhook from Admission requires it — Registration rejects without it.

Integration Challenges
10 / Challenges
10
Why Integration Is Hard
The real technical barriers we encountered when connecting three independently built systems
10
Technical Challenges
Each challenge required a deliberate design decision to solve
1

Different Tech Stacks

Laravel (PHP) · Vanilla PHP · React (JS). Different routing, auth, and session formats. Every boundary needs translation.

2

Three Separate Auth Systems

Admission: custom guard · Registration: Laravel Roles · Scheduling: custom session. Logged into one = stranger to the others. True SSO requires rewriting all three.

3

CORS Restrictions

Browsers block cross-domain requests. React (3000) vs systems (80) = blocked by default. Needed CORS middleware on each Laravel backend.

4

iFrame Security (X-Frame-Options)

Browsers block sites from loading inside iFrames by default. Required AllowFraming middleware — a deliberate security trade-off.

5

No Shared Student Identity

Admission: gmail_account · Registration: student_number · Scheduling: user_id. No universal key links all three systems.

6

Webhook Reliability

No guarantee the webhook succeeds. If Registration is down, data is lost. Production needs a message queue (Redis / RabbitMQ) to retry failures.

Database Challenge
11 / DB
11
Why Centralizing the Database Is Difficult
The proposal called for a centralized database — here's why that's harder than it sounds
11
The Centralized Database Problem
Five specific technical barriers — and our pragmatic solution
Problem 1

Table Name Conflicts

All 3 systems have a users table — can't coexist with the same name. Fix: prefixes → adm_ · reg_ · sch_.

Problem 2

Different Database Engines

MySQL (Admission + Registration) vs PostgreSQL (Scheduling cloud). Different SQL syntax — can't force both onto one engine.

Problem 3

Live Migration Risk

Moving live data risks data loss, downtime, broken references. Needs a full maintenance window — not feasible mid-semester.

Problem 4

Incompatible Data Models

Admission: gmail_account · Registration: email · Scheduling: numeric user_id. Merging needs a new universal model + full rewrite.

Problem 5

Multi-Team Ownership

3 teams, 3 codebases. Shared DB = shared failure point. One bad query corrupts everyone's data.

Our Solution

Partial Sharing + Prefixes

Admission & Registration share university_db with prefixes. Scheduling keeps its own DB. Practical centralization without a full rewrite.
